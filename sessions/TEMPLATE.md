# セッション作成テンプレート

このドキュメントは、Session 01 以降のセッションを作成する際に従うべき統一フォーマットを定義しています。

---

## 📐 セッション全体の構成

```markdown
# Session XX: セッションタイトル～サブタイトル

## 📌 このセッションで学ぶこと

- 学習目標1
- 学習目標2
- 学習目標3

---

## 🎯 セッション概要

| ステップ | 目的 | 所要時間 |
|---------|------|---------|
| ステップ N | 説明 | X 分 |

---

## ステップ N: ステップ名

### 🎯 目的

ステップの目的を 1-2 文で説明。参加者が「何をするのか」が明確に分かること。

### 💬 参加者の操作

```
1. 操作1
   ↓
2. 操作2
   ↓
3. 結果
```

**ポイント：** テキストフローを使用。矢印 `↓` または `→` で操作の流れを可視化。

### 📚 何が起きているのか（または「何を学ぶのか」）

- 背景説明や技術的な詳細
- なぜこの操作が重要なのか
- 参加者が期待すべき結果

### 💡 学習ポイント

- **キーポイント1：** 簡潔に説明
- **キーポイント2：** 簡潔に説明
- 🚀 **重要な気づき：** 参加者が得るべき重要な理解

### 📝 期待される結果 / 回答例（必要に応じて）

```
コード例やテキスト出力の例を表示
```

### 🎲 追加チャレンジ（オプション）

発展的な質問や実験を提案。より深く学びたい参加者向け。

---

## ✅ セッション完了チェック

- [ ] チェック項目1
- [ ] チェック項目2
- [ ] チェック項目3

---

## 📌 次のセッションへ向けて

次のセッション（Session XX）では～

を学びます。
```

---

## 📋 セクション名の役割と使い分け

| セクション名 | 役割 | 使用タイミング |
|-----------|------|------------|
| `🎯 目的` | そのステップの目標を明確に | 全ステップで必須 |
| `💬 参加者の操作` | 参加者が実際にやることを手順化 | 全ステップで必須 |
| `📚 何が起きているのか` / `何を学ぶのか` | ステップの背景・技術説明 | 全ステップで必須 |
| `💡 学習ポイント` | ステップから得るべき学習内容 | 全ステップで必須 |
| `📝 期待される結果 / 回答例` | 参加者が期待すべき出力・結果 | 結果がある場合のみ |
| `🎲 追加チャレンジ` | 発展的な質問・実験 | オプション（深掘り用） |

---

## 🎨 アイコン・絵文字使用ルール

セッションを視覚的に理解しやすくするため、以下のアイコンを統一的に使用します。

| 用途 | アイコン | 例 |
|-----|--------|-----|
| 目的・ゴール | 🎯 | `### 🎯 目的` |
| 参加者の操作 | 💬 | `### 💬 参加者の操作` |
| 学習内容・説明 | 📚 / 📖 | `### 📚 何が起きているのか` |
| 学習ポイント | 💡 | `### 💡 学習ポイント` |
| 期待される結果 | 📝 | `### 📝 期待される結果` |
| チャレンジ | 🎲 | `### 🎲 追加チャレンジ` |
| セッション完了 | ✅ | `## ✅ セッション完了チェック` |
| 次へ | 📌 | `## 📌 次のセッションへ向けて` |
| 重要 | ⚠️ | `⚠️ 注意：～` |
| ポイント | 🔍 / 🔎 | 注釈や詳細説明 |

---

## ✍️ 文体・トーンガイド

### 基本ルール

- **敬語を使わない：** 「です・ます調」ではなく、カジュアルで親しみやすく
- **簡潔に：** 短い文を意識。一文は 1 つの情報を
- **能動的に：** 「〜される」より「〜する」
- **具体的に：** 抽象的な説明より、実例を示す

### 例

```markdown
❌ 悪い例：
「Claudeが自動的にプロジェクト構造を解析されることになります」

✅ 良い例：
「Claudeが自動的にプロジェクト構造をスキャン」
```

## 📊 セッション概要表のルール

セッションの最初に、ステップ数・所要時間などを表で示します。

```markdown
## 🎯 セッション概要

| ステップ | 目的 | 所要時間 |
|---------|------|---------|
| ステップ 1 | XXX の方法を学ぶ | 5 分 |
| ステップ 2 | XXX を実装する | 15 分 |
| ステップ 3 | XXX をテストする | 10 分 |
```

**ポイント：**
- ステップ数は 2-5 個が目安（1 セッションは 20-40 分程度）
- 所要時間は実際の参加者ペースを想定
- 「目的」は動詞 + 内容（「学ぶ」「実装する」「テストする」など）

---

## 🔍 参加者の操作フロー表記

参加者が実際にやることを、見やすくフロー化します。

### パターン 1: 単純な直線フロー

```markdown
### 💬 参加者の操作

```
1. Step A を実行
   ↓
2. Step B を確認
   ↓
3. 結果が表示される
```
```

### パターン 2: 分岐がある場合

```markdown
### 💬 参加者の操作

```
1. チャットで質問を入力
   ↓
2. Claudeが回答
   ├─ 回答A が返ってきた場合 → ステップ X へ
   └─ 回答B が返ってきた場合 → ステップ Y へ
```
```

### パターン 3: 複数の並列操作

```markdown
### 💬 参加者の操作

```
【操作A】
1. ファイルA を開く
2. コードを確認

【操作B】
1. チャットで質問を入力
2. 結果を確認
```
```

## 💡 「学習ポイント」の書き方

### NG例

```markdown
❌
### 💡 学習ポイント
- Claudeが便利である
- プロジェクト構造が重要である
- コンテキストは大事である
```

### OK例

```markdown
✅
### 💡 学習ポイント

- 🚀 **効率性の向上：** 従来は 30 分かかった環境構築が数分で完了
- 🎯 **質問の精度：** コンテキストがあると、Claudeの回答精度が飛躍的に向上
- 📖 **自動ドキュメント：** AI が生成しても、実用的で正確な情報が得られる
```

**ポイント：**
- 各ポイントに **アイコン + 見出し** をつける
- **理由・効果** を具体的に説明
- 参加者が「実は〜だったのか」と気づく内容を

## ✅ チェックリストの作成

セッション終了時に、参加者が自己確認できるチェックリストを用意します。

### NG例

```markdown
❌
## ✅ セッション完了チェック

- [ ] 理解した
- [ ] 完了した
```

### OK例

```markdown
✅
## ✅ セッション完了チェック

- [ ] CLAUDE.md ファイルが生成されたことを確認した
- [ ] CLAUDE.md の 4 セクション（概要・コマンド・アーキテクチャ・ルート）をざっと読んだ
- [ ] Claudeに最低 1 つの質問をして、正確な回答が返ってくるのを確認した
- [ ] 「手探りではこれは時間がかかる」と実感できた
```

**ポイント：**
- 曖昧でなく、**具体的な行動** をチェック項目にする
- 参加者が「これをやったら完了」と判断できるように
- 3-5 項目が目安

---

## 🎬 Next セッション へのつなぎ方

セッション終了時に、次のセッションへの期待感を高めます。

```markdown
## 📌 次のセッションへ向けて

次のセッション（Session 02）では、このコンテキストを活用して：

- **XXX の方法** を学ぶ
- **YYY を実装** する実践的なステップ
- **ZZZ をテスト** する方法

を体験します。

💡 **このセッションで学んだコンテキスト活用が、次のセッションの基礎になります。**
```

**ポイント：**
- 次セッションの 3 つの学習内容を明示
- 「このセッションとのつながり」を示唆
- 参加者のモチベーション維持

---

## 📝 セッション作成のチェックリスト

新しいセッションを作成する際に、以下を確認してください。

- [ ] セッションタイトルが明確か？（サブタイトルでセッションの特徴を表現）
- [ ] 「このセッションで学ぶこと」が 2-3 個ある？
- [ ] セッション概要表に全ステップが記載されているか？
- [ ] 各ステップに 4 つの基本セクション（目的・操作・説明・ポイント）があるか？
- [ ] 「参加者の操作」がテキストフロー化されているか？
- [ ] 学習ポイントが「なぜ」「効果」を含むか？
- [ ] チェックリストが具体的な行動か？
- [ ] 次セッションへのつなぎがあるか？
- [ ] 文体がカジュアルで親しみやすいか？
- [ ] アイコンが統一されているか？

## 🎓 作成例：Session 02 テンプレート案

```markdown
# Session 02: コンテキストを活用した機能追加～実装とテストの初体験

## 📌 このセッションで学ぶこと

- CLAUDE.md コンテキストを活用して、実装効率を上げる方法
- Server Actions と Prisma を使った機能追加の基本フロー
- Playwright で E2E テストを書く方法

---

## 🎯 セッション概要

| ステップ | 目的 | 所要時間 |
|---------|------|---------|
| ステップ 1 | 新機能の仕様を Claudeに説明 | 5 分 |
| ステップ 2 | Claudeが提案するコード実装を確認 | 10 分 |
| ステップ 3 | 提案されたコードを自分で実装 | 15 分 |
| ステップ 4 | 簡単な E2E テストを書く | 10 分 |

---

## ステップ 1: 新機能をClaudeに説明する

### 🎯 目的

Claudeに新機能の要件を説明して、どのファイルにどんなコードを書けばいいのかを提案してもらいます。

### 💬 参加者の操作

```
1. チャットで以下のように入力：

   「チケット一覧にステータスフィルタを追加したい。
    'OPEN'、'IN_PROGRESS'、'DONE' に絞り込める機能。
    どのファイルを変更すればいいか？」

   ↓

2. Claudeが CLAUDE.md のコンテキストを使って提案

   ↓

3. 提案を読んで、実装方針を理解
```

### 📚 何が起きているのか

- Claudeがプロジェクトの構造を「知っている」ので、正確な実装箇所を即座に提案
- 手探りでファイルを探す時間がゼロ
- 実装前に全体像が見える

### 💡 学習ポイント

- 🎯 **コンテキストの力：** 曖昧な要件説明でも、Claudeは正確なファイル名を返す
- 💭 **思考の可視化：** 実装の「どこを変えればいいか」が明確になる
- ⚡ **時間短縮：** 従来なら 10 分調査が必要な情報が 30 秒で手に入る

---

## ✅ セッション完了チェック

- [ ] Claudeに新機能について質問した
- [ ] Claudeの提案を読んで、実装するファイルが特定できた
- [ ] 実装してみた
- [ ] テストを書いてみた

---

## 📌 次のセッションへ向けて

Session 03 では、この実装パターンを応用して、さらに複雑な機能を追加します。

~~~
```

## 📞 質問・相談

新しいセッションを作成する際に、以下の点で迷ったら：

1. **ステップ数が多い？** → セッションを分割（複数セッションに）
2. **ステップが短すぎる？** → 「追加チャレンジ」を充実させる
3. **文体が堅い？** → もっとカジュアルに、例を増やす
4. **チェックリストが曖昧？** → 「具体的な行動」に変更

---

**このテンプレートに従うことで、すべてのセッションが統一された、わかりやすい構成になります。**
